<objective>
Close UAT Gap 1 (Test 3): magic-link email send fails.

Root cause (confirmed live, see debug session): `signInWithOtp()` returns HTTP 429
`over_email_send_rate_limit`. The live project's custom Resend SMTP is NOT actually in force, so
Supabase uses its built-in mailer with a ~2-4 emails/hour project-wide cap, which is exhausted —
brand-new email addresses also get 429 immediately, ruling out the per-recipient 60s limit. The
application code (`src/actions/auth.ts`) is CORRECT; it faithfully surfaces the 429 as
`es.errors.sendLinkFailed`. **This is a live Supabase/Resend configuration fix, not a source change.**

Secondary latent issue (from debug notes): sender `onboarding@resend.dev` only delivers to the Resend
account owner; once the rate limit clears, non-owner recipients would hit a 500. The fix therefore
also moves the sender to a verified custom domain.

Purpose: Restore the magic-link entry path (the second of the two ways INTO the app). Unblocks
re-testing UAT Test 4 (and, with Plan 06, the full phase).

Output: Custom Resend SMTP enabled + sender on a verified domain + raised email rate limit on the
LIVE project, and `supabase/config.toml` updated to document that configuration as the local source
of truth. No application code is changed — doing so would pretend to fix a config problem.
</objective>
