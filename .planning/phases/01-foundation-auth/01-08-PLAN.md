<objective>
Build the data layer for typed-invite-code entry. After this plan, the database can resolve a trip id from a short human-typed code (case-insensitive), the seed trip carries a typeable code, and the TypeScript surface knows about both. This unblocks the entry-UI slice in Plan 01-09.

Purpose: The re-scope (2026-06-01, see entry-model-invite-code note) moves v1 entry from magic-link to a typed hybrid code. The existing `invite_token uuid` is not typeable, so a parallel, typeable `invite_code` column plus a matching SECURITY DEFINER resolver are required before any entry UI can be built. This mirrors the existing `get_trip_id_by_invite_token` pattern from Plan 01-06 exactly (the [[anon-join-architecture]] 4-coordinated-fix chain stays intact — only the resolution input type changes uuid→text).

Output: One new migration adding the column + resolver, an edit to the seed migration giving the seed trip a typeable code, and regenerated/hand-edited database types. No application code changes here — Plan 01-09 wires the entry UI to this layer.

Decision — fate of invite_token: KEEP it. The column stays `NOT NULL DEFAULT gen_random_uuid()` and the old `get_trip_id_by_invite_token` resolver stays in place untouched. invite_token becomes vestigial for v1 entry but dropping it would require editing the already-shipped initial_schema + 01-06 migrations and risks the seed row. The cost of keeping a vestigial uuid column is zero; the cost of dropping it is regression risk on shipped, verified migrations. Phase 2 (trip creation) will populate invite_code going forward; invite_token can be formally retired in a later cleanup if desired.
</objective>

```sql
CREATE OR REPLACE FUNCTION public.get_trip_id_by_invite_token(lookup_token uuid)
RETURNS uuid
LANGUAGE sql
SECURITY DEFINER
STABLE
SET search_path = public
AS $$
  SELECT id FROM public.trips
  WHERE trips.invite_token = lookup_token;
$$;
GRANT EXECUTE ON FUNCTION public.get_trip_id_by_invite_token(uuid) TO anon, authenticated;
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous browser → get_trip_id_by_invite_code RPC | Untrusted typed code crosses here; resolved by a bounded SELECT, returns only a trip id |
| RPC (SECURITY DEFINER) → trips table | Controlled RLS bypass for code→id resolution only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-08-01 | Information Disclosure | get_trip_id_by_invite_code | mitigate | Function returns ONLY the trip id (uuid), never trip content. Reading any trip data still requires membership via the unchanged trips SELECT policy. Possession of a valid code IS the intended invite capability. Mirrors T-01-06-01. |
| T-01-08-02 | Elevation of Privilege | SECURITY DEFINER function | mitigate | SET search_path = public blocks schema-injection (same hardening as is_trip_member / get_trip_id_by_invite_token). Single bounded SELECT by normalized equality; no dynamic SQL, no writes. |
| T-01-08-03 | Spoofing / brute-force | Short invite codes are guessable (4-char suffix from an ~31-symbol unambiguous alphabet ≈ 31^4 ≈ 9.2e5 combinations per name-prefix) | mitigate (partial) — flagged for Plan 01-09 + Phase 5 | The resolver itself does not rate-limit. Membership grants read access to trip docs containing PII (CLAUDE.md privacy constraint), so enumeration is a real concern. v1 acceptance: the close-circle threat model + Supabase's project-level signInAnonymously rate limit (~30/hr/IP, gating the only way to act on a guessed code) reduce practical brute-force. RECORDED for Plan 01-09: do not echo whether a code maps to a real trip beyond the generic invalidJoinToken message (no oracle distinguishing "wrong format" from "valid format, no trip"). Phase 5 may add per-IP rate limiting on the join path + longer suffixes if abuse appears. Disposition: accept for v1 with documented mitigations; this is NOT a HIGH blocker because no PII is disclosed by resolution alone (only a uuid) and acting on a guessed code still requires the rate-limited anon-join path. |
| T-01-08-04 | Tampering | RLS policy regression | accept | No RLS policy is added or altered; the membership gate on trips is preserved verbatim. Verified by the acceptance grep (0 policy statements). |
| T-01-08-05 | Data integrity | invite_code uniqueness collision | mitigate | UNIQUE constraint on invite_code rejects duplicates at the DB level. Phase 2 trip creation must retry-on-unique-violation when generating codes (tracked in invite-code-schema todo). |
</threat_model>
