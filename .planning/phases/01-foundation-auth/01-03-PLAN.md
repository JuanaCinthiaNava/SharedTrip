<objective>
Ship the magic link auth vertical slice: user types email on `/` → form submits to server action → Supabase + Resend SMTP send a Spanish email with an anti-Gmail-threading subject → user taps link → `/auth/callback` exchanges PKCE code for a cookie session → user lands inside the seed trip from Plan 02 (`/t/{seed-trip-id}/docs` if they have membership, otherwise `/`). Output: A working magic-link sign-in flow. After this plan, the welcome page is functional — a real user can sign up, click the email link, and be authenticated. Plan 04 then adds the no-email anonymous path; Plan 05 adds the shell + Perfil + sign-out.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Server Action (`sendMagicLink`) | Untrusted email input crosses; validated by Zod schema. |
| Server Action → Supabase Auth (`signInWithOtp`) | Server-only call; uses publishable key (anon-equivalent). |
| Supabase Auth → Resend SMTP | Outbound email; relies on Resend SPF/DKIM for spoofing protection. |
| Email link → Route Handler (`/auth/callback`) | Untrusted PKCE code + state; Supabase validates server-side. |
| Cookie storage → browser | Session JWT stored in HttpOnly Secure cookies by `@supabase/ssr`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | Magic link forwarded/shared to a third party | mitigate | Supabase OTP set to 15-min expiry (user_setup dashboard config). Single-use: Supabase invalidates the link after `exchangeCodeForSession` succeeds once. |
| T-03-02 | Information Disclosure | Session token leaked via localStorage XSS | mitigate | `@supabase/ssr` stores session in cookies, NOT localStorage. Cookies are HttpOnly by default (set by `@supabase/ssr`). XSS would need additional CSRF to access. |
| T-03-03 | Elevation of Privilege | PKCE code replay (attacker intercepts callback URL) | mitigate | PKCE code is single-use (Supabase invalidates after `exchangeCodeForSession`). Code is bound to the original `code_verifier` stored in the requester's cookie — replay from a different browser fails (this is the same mechanism that causes Pitfall #3 iOS issue — security trade-off accepted). |
| T-03-04 | Spoofing | Phishing email impersonating SharedTrip magic link | accept | Personal-circle product; users are friends of the developer. Resend domain DNS (SPF/DKIM) reduces but does not eliminate phishing risk. Friend-circle trust model accepts this. |
| T-03-05 | Information Disclosure | Magic link URL logged in Vercel access logs (contains `?code=...`) | mitigate | Codes are single-use + 5-min validity → log exposure is low-value. Vercel access logs are scoped to project owner. If risk grows, switch callback to POST + body, but Supabase pattern is GET-based. |
| T-03-06 | Tampering | A malicious `next` query param redirects user to attacker URL after auth | mitigate | The callback route handler MUST validate that `next` is a relative path (starts with `/` and does not contain `//`). | T-03-07 | Repudiation | User claims they did not request a magic link | accept | Resend retains delivery logs for 7 days (free tier). Sufficient for friend-circle dispute resolution. |
| T-03-SC | Tampering | Resend SDK supply-chain (if used) | n/a | Plan 03 does NOT install the Resend SDK — the simple approach uses Supabase's built-in email engine via Resend SMTP. No new package installed. |
