## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → removeMember/leaveTrip | Untrusted tripId / targetUserId cross into the server |
| server → Supabase | DELETE authorized by RLS, not by app-layer role checks |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-11 | Elevation of Privilege | removeMember called by a non-admin | mitigate | The `trip_members` DELETE RLS policy authorizes ONLY admin-removes-member or self-leave; a forged removeMember call by a non-creator is denied at the DB regardless of UI — actor identity from `getUser()`, plain SSR client (no service-role bypass) |
| T-02-12 | Spoofing | actor identity in leave/remove | mitigate | Actor derived from server `getUser()`; the target id is only a row key, never grants the caller the target's authority |
| T-02-13 | Tampering | over-broad delete | mitigate | Plain SSR client under RLS (NOT service-role); delete is scoped `.eq('trip_id', tripId)` + the user/target id; cannot affect other trips |
| T-02-14 | Denial of Service | removed member lingering on screen | accept | D-13: removed/left member is bounced on their NEXT navigation via the layout null-trip guard; no live push (Realtime is Phase 4). Acceptable for a <10-person trip. |
</threat_model>
