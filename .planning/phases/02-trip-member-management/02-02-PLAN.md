## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → /trips/new | Untrusted form body (name/dates/desc) crosses into the server |
| client → createTrip | created_by / user_id must come from server session, never the body |
| server → Supabase (service-role) | Privileged create bypasses in-request RLS chicken-and-egg |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02 | Spoofing | createTrip `created_by` / creator `user_id` | mitigate | Identity taken ONLY from server `getUser()`/`signInAnonymously()` — request body supplies name/dates/description only, never an id (RESEARCH Pattern 1) |
| T-02-03 | Information Disclosure | SUPABASE_SECRET_KEY | mitigate | Service-role key read from `process.env.SUPABASE_SECRET_KEY` inside a `'use server'` file only; never `NEXT_PUBLIC_*`, never reaches the client bundle |
| T-02-04 | Tampering | service-role scope creep | mitigate | Service-role used ONLY for the create path's two bounded inserts (trips + creator membership) with server-derived identity; edit/delete (02-05) run under normal RLS |
| T-02-05 | Elevation of Privilege | invite_code guessability | accept | Display capability code; unambiguous-alphabet entropy + DB UNIQUE constraint; possession-equals-capability matches the shipped resolver threat model (RESEARCH A1, V6 n/a) |
| T-02-06 | Information Disclosure | input validation | mitigate | Zod schema (name 1–80, end>=start) on the form; server may re-validate name length in the route handler before createTrip |
</threat_model>
